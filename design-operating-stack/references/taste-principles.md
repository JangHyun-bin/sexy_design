# Taste Principles

These principles apply to study, audit, generate, and systemize modes.

## 1. Object First

Every strong output needs a dominant object. If the product has no physical
object, create a typographic or operational object.

Good objects:

- product render
- phone surface
- card stack
- chart or timeline
- alarm clock or timer
- oversized amount
- engineered wordmark
- dashboard surface
- hand-drawn diagram

Weak objects:

- abstract gradient blob
- generic floating card
- fake browser frame
- stock illustration
- three feature cards with icons

## 2. One Dominant Gesture

Each first viewport gets one primary move.

Allowed gestures:

- giant wordmark
- oversized number
- cinematic product fold
- single flood color
- object hero
- paper or diagram metaphor
- phone-native state screen

If two gestures compete, remove one.

## 3. Accent Footprint Discipline

Accent color is assigned a footprint:

- `pin`: below 3 percent of viewport
- `signal`: 3 to 10 percent
- `field`: 10 to 35 percent
- `flood`: above 35 percent

Do not use accent color casually. A rust flood, red transfer button, orange task
pin, or yellow alarm field are valid because each has a job.

## 4. Material Surface

A surface must carry meaning.

Useful surfaces:

- grain for cinematic atmosphere
- halftone for technical darkness
- dot matrix for state or kinetic systems
- lined paper for academic or notebook metaphors
- industrial gray for product object authority
- frost or blur for finance depth
- chart grids for operational evidence

Avoid surfaces that are only mood.

## 5. Type As Product

Typography is not decoration. It can be the product's proof.

Useful roles:

- editorial serif for institutional trust
- giant grotesque for industrial confidence
- condensed technical sans for engineered products
- mono annotation for precision
- oversized numeric for finance, time, and status
- engineered wordmark for brand-first pages

Generic neutral sans hierarchy is not enough unless the object does the work.

## 6. Operational Sexiness

Show work happening.

Good evidence:

- transactions
- workflow runs
- charts
- specs
- timers
- alarms
- card states
- task tables
- control surfaces

Bad evidence:

- decorative chart with no encoded meaning
- fake metric
- testimonial without source
- screenshot used only as wallpaper

## 7. Mobile Is Not A Shrunk Website

Mobile has its own grammar.

Mobile work must consider:

- thumb zones
- bottom actions
- pressed states
- input focus states
- sheets and card depth
- oversized numbers
- safe areas
- dynamic type
- tactile controls
- state transitions

If a mobile screen is just a stacked desktop section, it fails.
